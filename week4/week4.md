## Content Insights

This week I learned that local development is different from simply editing a website online. A local development environment allows me to build and test a website on my own computer before making it public. This is useful because changes can be tested safely without affecting the live version of the website.

I also learned that a dynamic website needs more than just HTML and CSS files. Unlike the static website from Week 1, WordPress uses a database to store content such as pages, posts, settings, users, and plugin data. This helped me understand why tools like LocalWP are useful, because they automatically set up the server environment, WordPress files, and database together.

One important concept I learned is the difference between `localhost` and a production website. A localhost website only runs on my computer, so other people cannot access it through the internet. A production website is hosted online and can be visited by users through a public URL. This means local development is mainly used for building and testing, while production is used for the final public website.

I also learned that deployment is not always simple. A WordPress website includes both files and a database, so moving it from local development to production is more complicated than uploading a static HTML website. This made me realise that WordPress deployment needs careful attention to database export/import, file paths, plugins, themes, and site URLs.

For this practical, screenshots are important evidence because they can show that the site is running locally. Useful screenshots include:

* The WordPress site open in the browser with a localhost URL
* The LocalWP dashboard showing the local site running
* The WordPress admin dashboard
* Pages or plugins used for the startup website

This helped me understand that technical work often needs clear evidence, not just a final link.

## Career/Employability/Learning Insights

This week helped me understand that local development is an important professional workflow. In real web development, it is risky to make major changes directly on a live website. A local environment allows developers to test updates, plugins, layouts, and fixes before deployment. This reduces the chance of breaking a public website.

I also realised that CMS development is more complex than I first thought. In Week 1, publishing a static website was mostly about HTML files and GitHub Pages. With WordPress, the website depends on the CMS, server environment, themes, plugins, and database. This made me understand why employers value people who understand both website design and deployment workflows.

This practical also showed me that I need to become more comfortable with technical setup. Setting up a local environment felt confusing at first because there were more parts involved, such as the local server, WordPress installation, database, admin account, and browser URL. However, once I understood the purpose of each part, the process became more logical.

I noticed that my learning works better when I follow the practical steps first, then reflect on what each tool is actually doing. If I only follow instructions without understanding the reason behind them, I can complete the task but may not be able to fix problems later. For future weeks, I should spend more time understanding the workflow instead of only trying to finish quickly.

From an employability perspective, I think local development and deployment are useful skills because they connect classroom website tasks to real workplace practice. A person who can build a website locally, test it properly, and prepare it for deployment is more useful than someone who only knows how to edit pages online.

This week also reminded me that I should keep improving my startup website gradually. Instead of treating each practical as a separate task, I can reuse and improve the same website each week. This will make the final assignment stronger because the content, design, CMS setup, and technical workflow will develop over time.
